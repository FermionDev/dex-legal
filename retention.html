DEX Data Retention and Deletion Policy

Effective Date: April 26, 2026 · Last Reviewed: April 26, 2026 · Review Cadence: at least annually, and following any material change to a connected third-party integration.

1. Scope and Architecture

DEX is a local-first desktop application developed and operated by FermionDev. All user data, including data obtained through Plaid, QuickBooks Online, and other integrations, is stored exclusively on the user's own device under the application's local data directory (e.g., data/ within the DEX installation). FermionDev does not operate servers that ingest, store, or process user data. This architectural fact governs the retention model below.

2. Categories of Data and Retention Periods

Data Category Storage Location Retention Period
Plaid access tokens (per Item) Local device — data/plaid-state.json Retained until user disconnects the Item or wipes Plaid data; access token is revoked via Plaid's /item/remove endpoint at deletion time.
Transaction data imported from Plaid Local device — DEX ledger Retained until user wipes Plaid data or deletes the local data directory.
QuickBooks Online OAuth tokens and synced data Local device Retained until user disconnects the integration or deletes the local data directory.
User-generated content (notes, calendar, contacts, etc.) Local device Retained until user deletes the entry or the local data directory.
Analytics, telemetry, server-side logs Not collected N/A — DEX collects no analytics or telemetry.

3. Deletion Mechanisms

DEX implements three independent deletion mechanisms, each enforced in code:

  • Per-Item disconnect. Removing a single connected institution: DEX calls Plaid's /item/remove endpoint to revoke the access token, then removes the item record from local state. (Implemented in server/plaid.cjs — disconnectItem().)
  • Wipe Plaid data. Removing all Plaid-sourced data: DEX iterates every connected Item, calls /item/remove for each access token, then purges every Plaid-sourced transaction from the local ledger and clears the Plaid state file. (Implemented in server/plaid.cjs — wipe().)
  • Full local deletion. Removing all DEX data: the user deletes the application's local data directory and/or uninstalls DEX. Because no data exists on FermionDev infrastructure, this constitutes complete deletion.

User-facing instructions for each mechanism are published at /delete-account.html.

4. User Deletion Requests

Because data is stored locally on the user's device, users execute deletion themselves using the mechanisms above; no request to FermionDev is required. Users may nevertheless contact tfarkas777@gmail.com for assistance, and FermionDev will respond within 30 days. Verified deletion requests under GDPR, CCPA, PIPEDA, or other applicable frameworks are honored.

5. Third-Party Provider Deletion

Disconnecting a third-party integration in DEX revokes credentials and tokens with that provider as described in §3. Data the user originated within those providers (e.g., the user's bank, QuickBooks Online) remains under the provider's control and is governed by that provider's retention policy. Users may also revoke DEX's access directly through each provider's account settings.

6. Compliance

This policy is designed to be consistent with the deletion and retention requirements of GDPR (Art. 17), CCPA/CPRA, PIPEDA, and similar frameworks. Because FermionDev does not retain user data on its own infrastructure, the user retains full and immediate control over their data lifecycle. FermionDev does not sell user data and does not share user data with advertisers or data brokers.

7. Periodic Review

This policy is reviewed at least annually by FermionDev, and additionally upon any of the following triggers:

  • Addition or removal of a third-party integration.
  • Material change to a connected provider's API surface or data-handling terms.
  • Change in applicable privacy law that materially affects retention obligations.

The "Last Reviewed" date at the top of this page reflects the most recent review.
